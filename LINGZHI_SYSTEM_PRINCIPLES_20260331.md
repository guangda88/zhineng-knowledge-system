## 一、核心理念

### 1.1 根本原则

> **"注重实践，避免空谈，一切围绕用户生命状态的提升提供服务"**

这是系统的最高准则，所有技术、功能、设计都必须服务于这个目标。

### 1.2 系统定位

**灵知系统** = 集科学研究、理论探索、实践指导于一体的智能生命状态提升系统

**不是**：
- ❌ 简单的知识问答系统
- ❌ 技术展示平台
- ❌ 学术研究工具

**而是**：
- ✅ 帮助用户提升生命状态的智能辅助系统
- ✅ 连接科学、理论、实践的完整知识体系
- ✅ 个性化、尊重用户意愿的成长伙伴

### 1.3 核心价值观

1. **知行合一**
   - 理论理解 + 实践应用
   - 科学验证 + 生命改变

2. **用户中心**
   - 尊重用户意愿
   - 个性化服务
   - 不死板、不教条

3. **技术服务生命**
   - 技术是手段，生命是目的
   - 技术提升 → 服务更好 → 生命改变
   - 两者统一，不矛盾

4. **完整知识体系**
   - 🔬 科学研究：发现规律，提供依据
   - 📚 理论探索：深化理解，指导实践
   - 🎯 实践指导：具体方法，可操作步骤
   - 三者相互促进，缺一不可

## 二、系统能力架构

### 2.1 五大核心能力

#### 1️⃣ 智能检索
**目的**: 让用户快速找到需要的知识

**技术手段**:
- PostgreSQL + pgvector（向量数据库）
- bge-small-zh-v1.5（512维嵌入）
- HNSW索引（高性能检索）
- BM25（关键词检索）
- 混合检索（向量 + 关键词）

**为生命服务**:
- 3秒内找到理论知识
- 3秒内找到实践方法
- 3秒内找到科学依据
- **降低实践门槛，快速开始**

---

#### 2️⃣ 自学习与自进化
**目的**: 让系统越来越智能，服务越来越好

**技术手段**:
- GitHub监控：跟踪前沿技术
- 自主搜索：遇到难题自动查询
- 实验验证：分支测试，通过后合并
- 定期调度：自动化学习

**为生命服务**:
- 学习更好的技术 → 检索更准 → 用户更快找到方法
- 跟踪科学研究 → 提供最新依据 → 增强用户信心
- 发现有效方法 → 推荐给用户 → 提升实践效果
- **系统进步 = 服务质量提升**

---

#### 3️⃣ 内容生成
**目的**: 降低学习门槛，提供多样化支持

**技术手段**:
- 报告生成：学术报告、科学研究
- PPT生成：教学材料
- 音频生成：TTS语音
- 视频生成：教学视频
- 课程生成：完整课程体系

**为生命服务**:
- 🔬 科学研究报告：提供科学依据，增强信心
- 📚 学术综述：深化理解，精准练习
- 🎯 实践指南：具体方法，可直接操作
- 📅 个性化计划：从2天体验到5年规划
- **多样化内容 = 满足不同用户需求**

---

#### 4️⃣ 外部API集成
**目的**: 开放系统能力，服务更多用户

**技术手段**:
- REST API接口
- API密钥认证
- 基于角色的权限控制
- 速率限制保护

**为生命服务**:
- 第三方应用可以集成
- 更多用户可以受益
- 不同场景的灵活应用
- **服务扩展 = 生命影响扩大**

---

#### 5️⃣ 自优化（LingMinOpt框架）
**目的**: 系统持续改进，服务质量提升

**优化来源**:
- 系统报错：发现并修复问题
- 用户反馈：了解真实需求
- 审计结果：定期检查改进
- 性能指标：优化用户体验
- 学习洞察：应用新发现

**为生命服务**:
- 响应更快 → 用户体验更好
- 准确率更高 → 找到更相关的方法
- 界面更友好 → 降低使用门槛
- **技术优化 = 服务体验改进**

---

## 四、用户服务流程

### 4.1 完整的服务闭环

```
用户提问
    ↓
智能检索（理论 + 实践 + 科学）
    ↓
完整回答（知其然 + 知其所以然 + 知其所以必然）
    ↓
个性化计划（尊重用户意愿）
    ↓
实践支持（持续指导）
    ↓
效果追踪（生命状态改变）
    ↓
持续优化（越用越好）
```

### 4.2 问答响应结构

每个回答都应包含：

1. **理论理解**（是什么）
   - 基本概念
   - 历史渊源
   - 核心原理

2. **科学依据**（为什么有效）
   - 研究发现
   - 科学验证
   - 数据支撑

3. **实践方法**（怎么做）
   - 具体步骤
   - 注意事项
   - 预期效果

4. **个性化建议**（如何开始）
   - 根据用户水平
   - 根据用户时间
   - 根据用户目标
   - **尊重用户意愿**

## 五、个性化服务原则

### 5.1 不死板、不教条

❌ **错误做法**：
- 所有人都用21天模板
- 固定的练习计划
- 一成不变的方法

✅ **正确做法**：
- 完全根据用户情况定制
- 从2天体验到5年规划
- 灵活调整，随时优化

### 5.2 计划类型多样性

| 计划类型 | 适用场景 | 时长 | 特点 |
|---------|---------|------|------|
| 体验计划 | 只是好奇 | 2天 | 轻松体验，无压力 |
| 习惯养成 | 建立习惯 | 7-21天 | 循序渐进，重在坚持 |
| 问题解决 | 具体问题 | 7天 | 针对性强，验证效果 |
| 中期提升 | 稳步提升 | 3个月 | 系统学习，深化体验 |
| 长期修行 | 深入修行 | 1-5年 | 层层递进，系统规划 |

### 5.3 生成个性化计划的流程

```
询问用户意愿
    ↓
了解用户情况（水平、目标、时间）
    ↓
生成初步计划
    ↓
用户确认/调整
    ↓
最终确定计划
    ↓
执行中可调整
```

**关键**: 每一步都尊重用户意愿

## 六、技术架构

### 6.1 技术栈

**后端**:
- FastAPI（Web框架）
- PostgreSQL + pgvector（数据库）
- Redis（缓存）
- SQLAlchemy（ORM）

**AI/ML**:
- bge-small-zh-v1.5（嵌入）
- OpenAI API（大语言模型）
- Tesseract/PaddleOCR（OCR）
- Whisper（语音识别）

**部署**:
- Docker + Docker Compose
- Nginx（反向代理）

### 6.2 技术与生命服务的关系

| 技术 | 技术指标 | 生命服务指标 |
|------|---------|-------------|
| 向量检索 | 响应时间<100ms | 用户3秒找到方法 |
| 内容生成 | 生成速度<5s | 用户获得清晰指导 |
| 系统优化 | 准确率>95% | 用户找到正确方法 |
| 用户体验 | 界面友好 | 降低练习门槛 |

**结论**: 技术指标服务于生命指标

## 七、成功指标

### 7.1 技术指标（手段）

- API响应时间
- 检索准确率
- 系统稳定性

### 7.2 生命指标（目的）

- **实践转化率**: 知道理论后开始实践的比例
- **21天坚持率**: 持续练习21天的用户比例
- **生命状态改善率**: 自报告有改善的用户比例
- **推荐意愿**: 推荐给朋友的比例

### 7.3 两者关系

```
技术指标提升 → 用户体验改善 → 生命指标提升
```

**技术指标不是目的，是手段**

## 八、敏感词策略

### 8.1 分层策略

1. **公开层（未登录）**
   - 使用中性术语
   - 避免敏感词汇
   - 提供基础服务

2. **用户层（已登录）**
   - 可以显示更多内容
   - 保持学术真实性
   - 提供完整服务

3. **知识检索层**
   - 完全保持学术真实性
   - 不因敏感词删改内容
   - 知识完整性优先

### 8.2 目的

- 避免不必要的麻烦
- 保持学术完整性
- 平衡安全与真实

## 九、开发准则

### 9.1 功能设计三问

每个功能设计时必须问：

1. **这个功能如何帮助用户实践？**
2. **如何验证它真的改善了用户生命状态？**
3. **成功指标是什么？（包括技术指标和生命指标）**

### 9.2 代码审查标准

每次代码审查时检查：

- ✅ 是否为生命状态提升服务？
- ✅ 是否尊重用户意愿？
- ✅ 是否有科学依据？
- ✅ 是否可验证效果？

### 9.3 定期审计

- **每月**: 代码审计，检查是否符合核心原则
- **每季度**: 效果评估，检查生命指标
- **每年**: 战略回顾，调整发展方向
