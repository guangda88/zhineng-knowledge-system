## 二、通用工程流程

### 2.1 开发循环

```
需求 → 分支 → 编码 → 自检 → 提交 → PR → CI → 合并 → 部署
                  ↓                              ↑
            pre-commit hooks               GitHub Actions
         (black/isort/flake8/bandit)    (lint/test/security)
```

### 2.2 分支策略

```
main (生产, 受保护)
  └── develop (集成)
        ├── feature/xxx
        ├── fix/xxx
        └── hotfix/xxx
```

- **main**: 生产环境，仅通过 PR 从 develop 合并
- **develop**: 开发集成，日常开发基准
- **feature/***: 新功能，从 develop 创建
- **fix/***: Bug 修复，从 develop 创建
- **hotfix/***: 紧急修复，从 main 创建，合并回 main + develop

### 2.3 提交规范

```
<type>(<scope>): <subject>

Types: feat, fix, docs, style, refactor, perf, test, chore, revert
Scopes: backend, frontend, api, db, auth, docs, ci
```

### 2.4 CI/CD 流水线

| 阶段 | 工具 | 门槛 | 超时 |
|------|------|------|------|
| **lint** | flake8 | 零 error | 10 min |
| **test** | pytest + PostgreSQL + Redis | 覆盖率 ≥ 60% | 20 min |
| **security** | bandit | advisory | 15 min |
| **status-check** | 聚合 lint+test | 全部通过 | — |

## 三、项目原则

### 3.0 核心原则（最高准则）

> **"注重实践，避免空谈，一切围绕用户生命状态的提升提供服务"**

这是系统的最高准则，所有代码、功能、架构决策都必须服务于这个目标。

#### 项目定位

**灵知系统** = 集科学研究、理论探索、实践指导于一体的智能生命状态提升系统

#### 核心价值观

1. **知行合一**
   - 理论理解 + 实践应用
   - 科学验证 + 生命改变

2. **用户中心**
   - 尊重用户意愿
   - 个性化服务
   - 不死板、不教条

3. **技术服务生命**
   - 技术是手段，生命是目的
   - 技术提升 → 服务更好 → 生命改变
   - 两者统一，不矛盾

4. **完整知识体系**
   - 🔬 科学研究：发现规律，提供依据
   - 📚 理论探索：深化理解，指导实践
   - 🎯 实践指导：具体方法，可操作步骤

#### 核心原则三问

**所有功能设计、架构决策必须回答**：

1. **这个功能如何帮助用户实践？**
2. **如何验证它真的改善了用户生命状态？**
3. **成功指标是什么？（包括技术指标和生命指标）**

如果无法回答这三个问题，功能不应该开发。

#### 技术与生命服务

| 技术 | 技术指标 | 生命服务指标 |
|------|---------|-------------|
| 向量检索 | 响应时间<100ms | 用户3秒找到方法 |
| 内容生成 | 生成速度<5s | 用户获得清晰指导 |
| 系统优化 | 准确率>95% | 用户找到正确方法 |
| 用户体验 | 界面友好 | 降低练习门槛 |

**结论**：技术指标服务于生命指标

#### 标注系统

**目的**：提升图片和音视频的识别精度

**为生命服务**：
- ✅ OCR标注：用户学到正确的功法要领
- ✅ 转写标注：用户听到正确的口令词
- ✅ 视频标注：用户理解正确的动作要领

**核心价值**：
```
标注提升精度
    ↓
用户获得准确内容
    ↓
练习正确
    ↓
生命状态提升
```

#### 个性化服务原则

**不死板、不教条**：
- ❌ 错误：所有人都用21天模板
- ✅ 正确：从2天体验到5年规划，完全个性化

**计划类型**：
- 体验计划：2天轻松体验
- 习惯养成：7-21天建立习惯
- 问题解决：针对具体问题
- 中期提升：3个月稳步提升
- 长期修行：1-5年系统规划

#### 完整知识体系

每个回答都应包含：
1. **理论理解**（是什么）
2. **科学依据**（为什么有效）
3. **实践方法**（怎么做）
4. **个性化建议**（如何开始，尊重用户意愿）

### 3.1 架构原则

| 原则 | 说明 | 示例 |
|------|------|------|
| **异步优先** | 所有 I/O 使用 async/await | asyncpg, httpx.AsyncClient |
| **单例统一** | 连接池/Redis/Config 各只有 1 个权威实例 | `database.py` 拥有 `db_pool` |
| **委托优于复制** | 子模块通过委托访问权威实例 | `get_db_pool()` 委托给 `database.py` |
| **优雅降级** | 可选服务失败不阻塞启动 | 监控/领域/缓存均可选 |
| **线程安全** | 共享资源的初始化必须有锁 | `_db_pool_lock`, `_config_lock` |
| **配置外部化** | 所有可变配置通过环境变量 | `.env` + Pydantic Settings |
| **最小权限** | 生产环境默认拒绝 | 未配置 API Key → 401 |

### 3.2 安全原则

| 原则 | 说明 | 实现 |
|------|------|------|
| **不硬编码凭据** | 禁止代码中出现密码/Key | 环境变量 + 503 未配置时 |
| **日志脱敏** | URL/连接串中的密码不记录 | `_sanitize_url()` → `***` |
| **常量时间比较** | 密码/Token 使用 hmac.compare_digest | `authenticated.py`, `dependency_injection.py` |
| **请求体传 Token** | JWT refresh 使用 POST body | Pydantic `RefreshRequest` |
| **参数化查询** | 禁止字符串拼接 SQL | asyncpg `$1, $2` 参数 |
| **CORS 白名单** | 生产环境必须配置 ALLOWED_ORIGINS | JSON 数组或逗号分隔 |

### 3.3 代码质量原则

| 原则 | 说明 |
|------|------|
| **读后编辑** | 修改任何文件前必须先阅读 |
| **改后即测** | 每次修改后立即运行测试 |
| **精确匹配** | 编辑时原文必须字节级精确匹配 |
| **增量推进** | 一次一个逻辑变更 |
| **类型注解** | 所有公共函数必须有类型注解 |
| **Google docstring** | 所有公共函数必须有 `Args:`/`Returns:` 文档 |

## 四、关键架构决策记录

### ADR-001: 统一数据库连接池

- **背景**: 原有 3 个独立 asyncpg.Pool（DI、lifespan、database.py）
- **决策**: `database.py` 拥有唯一权威 `db_pool`，其他模块通过委托访问
- **原因**: 避免资源浪费、连接泄漏、配置不一致
- **日期**: 2026-03-31

### ADR-002: 统一 Redis 客户端

- **背景**: 原有 3 个独立 Redis 连接
- **决策**: `CacheService` 拥有唯一权威 Redis 客户端，其他模块通过 `get_cache_manager()` 访问
- **原因**: 与 ADR-001 同理
- **日期**: 2026-03-31

### ADR-003: asyncpg 原生 SQL（无 ORM）

- **背景**: 项目同时引入 asyncpg 和 SQLAlchemy
- **决策**: 核心读写使用 asyncpg 原生 SQL（`$1` 参数），SQLAlchemy 仅用于 ORM 查询（如 books）
- **原因**: 原生 SQL 性能更好、更可控；pgvector 操作原生支持
- **日期**: 2026-03-25

### ADR-004: Pydantic Settings 多继承配置

- **背景**: 配置分散在多个文件中
- **决策**: `Config(BaseConfig, DatabaseConfig, RedisConfig, SecurityConfig, LingZhiConfig)` 多继承
- **原因**: 配置关注点分离，同时保持单一访问入口 `get_config()`
- **日期**: 2026-03-25

### ADR-005: v2 API 复用 v1 Router

- **背景**: v1/v2 books.py 100% 代码重复
- **决策**: v2 删除 books.py，`v2/__init__.py` 直接导入 v1 的 router
- **原因**: 消除代码重复，单一维护点
- **日期**: 2026-03-31

### 6.1 生命指标测量框架（P2 - 待补充）

**目的**: 让"如何验证它真的改善了用户生命状态"可操作、可测量

**直接指标**（Direct Metrics）:
- 连续练习天数
- 用户等级迁移（入门→进阶→高级）
- 生命状态自评（1-10分，每周追踪）

**代理指标**（Proxy Metrics）:
- 实践转化率（知道理论后开始实践的比例）
- 21天坚持率（持续练习21天的用户比例）
- 生命状态改善率（自评有改善的用户比例）
- 推荐意愿（愿意推荐给朋友的用户比例）

**数据表结构**:
- `user_levels` - 用户等级表
- `life_state_tracking` - 生命状态追踪表
- `practice_records` - 练习记录表

**测量流程**:
```
功能上线 → 设定目标 → 数据收集 → 数据分析 → 效果验证 → 决策
```

详细内容参考: [DEVELOPMENT_RULES.md 第0.2节](../DEVELOPMENT_RULES.md#02-生命指标测量框架)

### 6.2 核心原则三问的否决案例（P3 - 积累型）

**示例1**: "纯技术指标优化功能"被否决

**背景**: 有人提出"优化向量检索算法，将响应时间从100ms降到80ms"

**核心原则三问**:
1. 这个功能如何帮助用户实践？ - ❌ 只是让系统更快，用户不会因此多练习
2. 如何验证它真的改善了用户生命状态？ - ❌ 无法验证
3. 成功指标是什么？ - ❌ 只有技术指标（响应时间）

**决策**: ❌ 否决，理由是纯技术优化，不直接为用户生命服务

**替代方案**: 如果用户反馈"搜索太慢，找不到想要的内容"，则优化有明确用户价值

---

**示例2**: "学术报告生成器"被部分否决

**背景**: 有人提出"生成学术报告，帮助用户深入研究理论"

**核心原则三问**:
1. 这个功能如何帮助用户实践？ - ⚠️ 可以深化理解，但不够直接
2. 如何验证它真的改善了用户生命状态？ - ⚠️ 无法直接验证
3. 成功指标是什么？ - ⚠️ 只有生成质量指标

**决策**: ✅ 有条件通过，但必须补充"实践指导生成"

**改进方案**:
- 不仅生成学术报告，还要生成"如何将理论应用于实践的指南"
- 追踪：生成报告后，用户是否开始实践？
- 验证：阅读报告的用户，生命状态是否改善？
